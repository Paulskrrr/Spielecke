## 🔴 Fix first — High severity

Ten issues a group will actually hit at the table: wrong winners, frozen tabs, a house-rule toggle that silently does nothing, and a card that corrupts turn order. No data loss and nothing security-critical, but each one breaks the specific promise a game makes to its players.

### Bomb detonation setTimeout freezes during iOS screen lock — fuse silently extends, breaking the spec's 'precise timer' promise
**`js/games/bomb.js:170`** · Bug

The detonation is a plain setTimeout with no deadline reconciliation. On iOS Safari the JS timer clock is frozen while the page is hidden (screen lock, app switch, incoming call). The fuse is 20-90s and iPhone default auto-lock is 30s; if nobody touches the screen for 30s mid-fuse (someone freezes, an argument starts), the screen locks, the timer stops, and after unlock the bomb keeps 'burning' for however much setTimeout budget was left — the round silently extends by the whole locked duration. PARTY-APP-SPEC.md:255-256 explicitly promises 'detonation on a precise timer'. Worse, the cosmetic tick scheduler at bomb.js:243-249 computes frac from Date.now() (wall clock, which does NOT freeze), so after wake `frac` clamps to 1 and the ticks race at the maximum 110ms panic rate signalling imminent explosion — while the actual detonation is still arbitrarily far away. No visibilitychange handler and no navigator.wakeLock request exists anywhere in js/ (grep confirms).

> **Evidence:** bomb.js:170 `fuseTimeout = global.setTimeout(detonate, fuseMs);` — and bomb.js:244-245 `var elapsed = Date.now() - start; var frac = Math.min(elapsed / fuseMs, 1);` with no visibility handling anywhere.

**Fix:** Record `var deadline = Date.now() + fuseMs` at arm time; add a `visibilitychange`/`pageshow` listener (torn down in unmount) that on becoming visible checks `Date.now() >= deadline` and calls detonate() immediately, else re-schedules setTimeout for the remaining wall-clock time. Additionally request `navigator.wakeLock.request('screen')` (supported iOS Safari 16.4+, try/catch, release in unmount) while a fuse is live so the screen doesn't lock mid-round in the first place.

### "Tie counts as wrong" house rule has no effect — ties always fail even when toggled off
**`js/games/busfahrt.js:221`** · Bug

judge() returns the sentinel "tie" ONLY when settings.tieWrong is false (lines 239, 245: `return settings.tieWrong ? false : "tie"`), i.e. when the house rule says a tie must NOT count as wrong. But guess() treats "tie" exactly like false and calls fail(), charging the driver sips and resetting the row. The setup toggle 'Tie counts as wrong (steps 2 & 3)' (line 93) therefore changes nothing observable: with the rule off, a tied card on step 2 or 3 still fails the ride and the driver still drinks.

> **Evidence:** js/games/busfahrt.js:221 `if (correct === "tie" || correct === false) { fail(); }` combined with :239 `if (cur === prev) return settings.tieWrong ? false : "tie";` and :245 `if (cv === lo || cv === hi) return settings.tieWrong ? false : "tie"; // on the line`

**Fix:** Since judge() only ever returns "tie" when the rule is OFF, treat it as a pass: change the check to `if (correct === false) { fail(); } else { step++; ... }` (optionally show a 'Gleichstand — frei' note in #bf-result before advancing).

### "Dreifacher Zug" card has no engine support — resolveCard unconditionally advances the turn, so the three mandated extra draws are attributed to the wrong players
**`js/games/hochadel.js:529`** · Bug

The deck contains s_dreifachzug ("Ziehe sofort drei weitere Karten — du bleibst am Zug." / "Draw three more cards at once — you remain at turn.", js/content/hochadel.js:212-216 and 449-450), but resolveCard() has no hook for it: after any card it calls nextTurn() and re-renders the table with the NEXT player as "Am Zug". If the table obeys the card and the same player draws three more cards, every one of those draws is resolved under the wrong currentPlayer(): fillName() bakes the wrong name into {P} rule texts (line 170), regel cards record the wrong `by` (line 517), aktiv cards record the wrong `holder` (line 526) permanently on the face-up trump, and the turn pointer ends up skipped three seats forward. The engine already has an effect hook for exactly this class of card (`if (card.effect === "reverse") game.dir = -game.dir;`, line 512), so this one card silently corrupts turn order and holder state in every session where it is drawn.

> **Evidence:** js/games/hochadel.js:512-531: `if (card.effect === "reverse") game.dir = -game.dir; ... nextTurn(); saveState(); renderTable();` — js/content/hochadel.js:212-213: `{ id: "s_dreifachzug", type: "sofort", title: "Dreifacher Zug", text: "Ziehe sofort drei weitere Karten — du bleibst am Zug.", ... }`

**Fix:** Add an engine hook mirroring `effect: "reverse"`, e.g. `effect: "extraTurn"` with a pending-extra-draws counter (n=3): in resolveCard, skip nextTurn() while extra draws remain and decrement per draw. Tag s_dreifachzug with it in both DE and EN decks.

*Verifier note: All file/line citations are accurate. Minor: severity "high" could be argued down to medium — the game stays playable and only misattributes/mis-advances when this single card is drawn — but the wrong holder/by records are persisted permanently and the turn pointer is corrupted, so high is defensible.*

### Liar's Numbers reveal has no tie handling — equal-distance players get arbitrary opposite outcomes
**`js/games/liars.js:122`** · Bug

renderReveal() sorts guesses by absolute distance only and declares ranked[0] the sole winner and ranked[last] the sole loser. Ties in distance (two players entering the identical number, or symmetric over/under guesses like 40 and 60 around 50) are broken by stable-sort insertion order, i.e. the shuffled pass order. Concrete failure: question 'How many countries are in the EU (2024)?' (a=27), both players guess 27 → screen shows '🏆 P1 nailed it!' and '🍺 P2 was furthest — drink!' with both rows reading '(off 0)'. Identical guesses are common with year/count questions in this content, so groups will hit this.

> **Evidence:** js/games/liars.js:122-126 `var ranked = guesses.slice().sort(function (x, y) { return Math.abs(x.guess - a) - Math.abs(y.guess - a); }); var winner = ranked[0]; var loser = ranked[ranked.length - 1];` and :139-141 `var loseLine = settings.drinking ? "🍺 <strong>" + esc(loser.name) + "</strong> " + t("was furthest — drink!") ...`

**Fix:** Group players by distance: everyone at the minimum distance shares the crown, only players at the strictly maximum distance drink, and if min === max (everyone tied) declare a wash with neither winner nor loser. Apply the same grouping to the 👑/🍺 row tags at line 130.

*Verifier note: Details are accurate as stated (file, lines 122-126, 130, 139-141). Severity 'high' is arguable for a party game — it's a fairness/UX bug, not a crash or data loss — 'medium' may be fairer, but the described behavior is real and common in play.*

### Horse Race: last session's persisted bets are silently counted as tonight's bets in the payout, and a bet can never be cleared
**`js/games/pferderennen.js:58`** · Bug

mount() loads the previous session's bets from 'spielecke.game.pferderennen.bets' and treats them as live bets: renderSetup pre-ticks the suit (line 90), and finish() builds payout rows from every roster player with an entry in `bets` (line 359: roster().filter(p => bets[p.id])). A player who bet hearts last week and doesn't touch their row tonight is ordered to 'trinkt 3 sips' (or 'verteilt') for a bet they never placed this race. The suit buttons only ever assign (line 129 `bets[player] = suit`) — there is no toggle-off and no reset — so once a player has ever bet, the stale bet is impossible to remove except by switching to a different suit; 'New bets 🎲' (line 392) just returns to the same pre-ticked setup. This is the direct sibling of the Hochadel stale-roster bug fixed in 5402921.

> **Evidence:** line 58: `bets = context.store.get("bets", {}) || {};`  line 90: `var on = bets[p.id] === s ? " hr-suit--on" : "";`  line 359: `var entries = roster().filter(function (p) { return bets[p.id]; }).map(...)`  line 129-130: `bets[player] = suit; ctx.store.set("bets", bets);`

**Fix:** Treat bets as per-session state: don't load them in mount() (start with `bets = {}` and stop persisting, or persist only within the mounted session). If pre-selection is wanted, require an explicit confirmation per row before a bet counts, and make tapping the active suit again deselect it (delete bets[player]) so a player can opt out.

### Rank It declares "perfect match — the whole table agrees" on any drift tie; 37% of 2-player rounds resolve wrongly with no winner/loser
**`js/games/rankit.js:271`** · Bug

renderReveal decides the no-win/no-lose "everyone agrees" outcome by comparing the best and worst drift scores, not by checking that the rankings are identical. Because the consensus (average-position order) sits symmetrically between rankings, tied footrule drifts are extremely common with different orders. Brute force over all 14,400 pairs of 5-item permutations (replicating this exact math): 37.2% of 2-player rounds tie, 36.4% with DIFFERENT orders — e.g. P1=[1,0,2], P2=[2,1,0] both get drift 2. So at the game's own minimum player count (MIN_PLAYERS = 2, line 19), over a third of rounds show "A perfect match — the whole table agrees! 🤝" (line 294) while the 🔍 compare view visibly shows two different rankings, no 👑/💀 tags are assigned (line 283), and in drinking mode nobody drinks. Monte Carlo shows it also hits ~4.6% of 3-player rounds. The inline comment states the intent ("everyone identical → no win/lose"), but the check tests drift equality, not order identity.

> **Evidence:** var split = winner.drift !== loser.drift; // everyone identical → no win/lose
...
var verdict = !split
  ? '<p class="result-sub">' + t("A perfect match — the whole table agrees! 🤝") + "</p>"

**Fix:** Compute identity explicitly, e.g. var allSame = scored.every(function (s) { return s.order.join() === scored[0].order.join(); }); use allSame for the "perfect match" verdict. For a mere drift tie, either crown/penalise all tied players (everyone with drift === winner.drift gets 👑, everyone with drift === loser.drift gets 🍺/💀) or show a neutral "dead heat" message — never claim the table agrees.

### pickName() infinite-loops and freezes the app when all roster names are identical
**`js/games/truth.js:89`** · Bug

The no-immediate-repeat guard compares by player NAME, not identity, using unbounded rejection sampling. roster.js addPlayer() (js/roster.js:81-87) does not prevent duplicate names, so a roster of e.g. two players both named "Max" passes the `roster.length === 1` early-return (length is 2), but after the first round `lastName === "Max"` and every sample satisfies `name === lastName` — the do-while never terminates and the main thread hangs; the tab must be killed. With partial duplicates (e.g. Max, Max, Anna) it also skews fairness: after any "Max" round only Anna is eligible, giving her ~50% of turns instead of ~33%.

> **Evidence:** js/games/truth.js:87-92: `if (roster.length === 1) return roster[0].name;\n    var name;\n    do { name = roster[Math.floor(Math.random() * roster.length)].name; }\n    while (name === lastName);` — and js/roster.js:81-87 `function addPlayer(name) { var trimmed = (name || "").trim(); if (!trimmed) return; players.push({ id: makeId(), name: trimmed }); ... }` (no duplicate check).

**Fix:** Track the last picked player's index or id instead of name, and pick from the filtered candidate list: `var candidates = roster.filter(function (p) { return p.id !== lastId; }); if (!candidates.length) candidates = roster; var p = candidates[Math.floor(Math.random() * candidates.length)]; lastId = p.id; return p.name;` — this terminates always and stays uniform over eligible players.

### Space-to-advance double-fires on focused tappable [data-primary] elements
**`js/shell.js:46`** · Bug

onGlobalKeydown never checks e.defaultPrevented. ui.js tappable() adds its own keydown handler for Space on the element (preventDefault + handler), and several games mark that same tappable element [data-primary]: nhie.js:73 (#ni-card), truth.js:76 (#tr-card), mostlikely.js:74 (#ml-card), maxchen.js:99 (#mx-hat), chooser.js:59 (#chooser-wheelwrap). Tappable elements get tabindex=0, so clicking one focuses it (Chrome/Firefox) or Tab reaches it. With the element focused, one Space press runs the handler twice: first via tappable's element-level keydown, then via the shell's window-level keydown which querySelects the same [data-primary] element and calls .click().

> **Evidence:** shell.js:46-57: `function onGlobalKeydown(e) { if (e.code !== "Space" ... var btns = contentEl.querySelectorAll("[data-primary]:not([disabled])"); ... btns[btns.length - 1].click(); }` — no e.defaultPrevented check; ui.js:30-35: `el.addEventListener("keydown", function (e) { if (e.key === "Enter" || e.key === " " || e.key === "Spacebar") { e.preventDefault(); handler(e); } });`

**Fix:** Add `if (e.defaultPrevented) return;` at the top of onGlobalKeydown (tappable already calls preventDefault, so the shell fallback then correctly stands down). Alternatively/additionally have tappable's keydown call e.stopPropagation().

*Verifier note: Severity is arguably medium rather than high: the bug is keyboard-only (requires the tappable element to be focused via prior click or Tab) on a mobile-first shared-screen app where most interaction is touch. Mechanics, files, and line numbers in the finding are otherwise accurate.*

### Router has no history integration: phone Back button/gesture exits the site and destroys the in-progress round
**`js/shell.js:107`** · UX

The router is purely in-memory: showShelf/showRoster/mountGame (js/shell.js:84-130) swap contentEl.innerHTML with no history.pushState, popstate, or hashchange handling anywhere in js/ (grep-confirmed). On the primary device per PARTY-APP-SPEC.md Part 0 ('a MacBook on the table or a phone passed around'), the hardware Back button (Android) or edge-swipe back gesture (iOS — trivially triggered by accident while the phone is handed around) does not return from a game to the shelf; it navigates the browser away from the site entirely. Because every game except Hochadel holds its round in module-local variables (see companion finding), the whole round is gone: a 45-minute Activity board race, all Quiz Out lives, half-collected Rank It rankings, a Busfahrt ladder run, a full Doodle Drama chain. Spec §1.5 requires 'Navigating into a game and back works cleanly' — on a phone the natural 'back' is the system gesture, which currently means 'leave and lose everything'. bfcache may sometimes restore on forward-swipe, but that is browser-dependent and any fresh load lands on the shelf (boot() at js/shell.js:23-29 always calls showShelf()).

> **Evidence:** js/shell.js:107-130 `function mountGame(meta) { teardownCurrent(); currentGame = meta.module; ... }` — no history API call anywhere; js/shell.js:23-29 `function boot() { ... renderHeader(); showShelf(); global.addEventListener("keydown", onGlobalKeydown); }` (the only global listener registered is keydown).

**Fix:** Push a history entry when entering a game/roster (history.pushState({screen:'game',id:...})) and handle popstate by routing to the shelf instead of leaving the page (works fine from file:// and GH Pages subpaths since it never changes the path, only state). First back-press from a game then returns to the shelf; a second back-press from the shelf exits the site as expected.

### One stray tap on the always-visible header instantly destroys the mounted game with no confirmation — most games have no recoverable state
**`js/shell.js:145`** · UX

renderHeader() wires #brand -> showShelf and #edit-players -> showRoster (js/shell.js:145-146); both call teardownCurrent() unconditionally (js/shell.js:63-73), which runs unmount() and wipes contentEl. The header is persistent on every screen (index.html:18), sits exactly where a phone is gripped when passed around, and there is no 'are you sure' guard while a game is mounted. Games whose unmount() irrecoverably discards an in-progress round (all state in module-local vars, none written to store): Activity — board positions live only in `teams[i].pos`, reset to 0 on every mount (js/games/activity.js:55-58; only fig0/fig1/teamAssign/drinking are persisted at :101/:108/:113), so a 45-minute board race restarts from square 0; Quiz Out — `players` (with lives), `roundQueue`, `round`, `used` are module vars cleared at js/games/quiz.js:49-52 (only the hearts/drinking *settings* persist at :43-44); Rank It — `rankings`, `idx`, `current` (the mid-collection private rankings already entered by some players) cleared at js/games/rankit.js:44-48; Ride the Bus — `driverIdx`, `row`, `step`, `busPos` (ladder progress and driver rotation) cleared at js/games/busfahrt.js:57-61, and driverIdx is reset to 0 on remount (:54); Doodle Drama — the entire drawing-telephone `chain` of canvases/guesses (js/games/doodle.js:24, cleared :42-45); Fuck the Dealer — `deck`/`discard`/dealer rotation cleared at js/games/fuckdealer.js:57-61. Only Hochadel survives, because it persists full game state to context.store (js/games/hochadel.js:66-70, saveState at :159) and reconciles on remount. Worse, the app itself invites mid-game header taps: quiz/rankit setup text says 'Add them from the header (👥)' — but tapping 👥 during a *running* round of any non-Hochadel game destroys it.

> **Evidence:** js/shell.js:145-146 `headerEl.querySelector("#brand").addEventListener("click", showShelf); headerEl.querySelector("#edit-players").addEventListener("click", showRoster);` and js/shell.js:63-73 `function teardownCurrent() { if (currentGame && ...) currentGame.unmount(); ... currentGame = null; contentEl.innerHTML = ""; }` — no confirmation branch when currentGame is set.

**Fix:** In showShelf/showRoster (or in the header click handlers), when `currentGame` is non-null ask for confirmation before teardownCurrent() — e.g. a one-tap confirm ('Leave game? The round will be lost.') skipped for games that persist (could be signalled via meta, letting Hochadel bypass it). Longer term, adopt the Hochadel pattern (persist round state via context.store) for the long-form games (Activity, Quiz Out, Busfahrt).

## Two structural risks worth calling out separately

These aren't single bugs — they're patterns that recur across most of the 20 games, surfaced by the completeness sweep rather than the per-game review.

**A round survives everything except leaving the screen.** Every game except Hochadel keeps its round in memory only. The router never touches `history.pushState`, so the phone's Back gesture exits the site instead of returning to the shelf. The always-visible header's home/players buttons tear down the mounted game with zero confirmation. A refresh has no `beforeunload` guard. (`js/shell.js:107`, `:145`, `:28`)

**iOS Safari quietly breaks timing and sound.** Three games rely on `setTimeout`/`setInterval` wall-clock timing and Web Audio, both of which iOS suspends on screen lock or the silent switch, with no `visibilitychange` recovery or `AudioContext.resume()` anywhere in the codebase. (`js/games/bomb.js:170`, `:271`, `js/games/whoami.js:202`)

**Persisted state doesn't track roster or language changes.** Only Hochadel was patched (commit `5402921`) to reconcile saved state against roster drift. Horse Race is the clearest sibling: last session's bets are silently replayed as tonight's bets, and a bet can never be cleared. (`js/games/pferderennen.js:58`, `js/games/activity.js:169`, `js/games/hochadel.js:66`)

**PARTY-APP-SPEC.md has drifted from the code.** Five sections still cite the old per-game minimum player counts after commit `781cd33` deliberately lowered them, and Hochadel's ground rules went from two to three without the doc — or the German UI string — catching up. (`PARTY-APP-SPEC.md:331`, `:407`, `js/i18n.js:407`)

## 🟠 Should fix soon — Medium severity

Real bugs and gaps a session will eventually hit, but that don't break a game outright — content repeats, i18n leaks, missed edge cases.

### Bug (18)

- **`js/shell.js:50`** — Space steals activation from any focused non-primary button and clicks the primary instead  
  _Fix:_ Bail out when the event target is itself an activatable control: `if (e.target && e.target.closest && e.target.closest('button, [role="button"]')) return;` before the querySelectorAll, letting the browser's native Space activation handle the focused control.
- **`js/shell.js:122`** — mountGame error path skips unmount, leaking whatever the game set up before throwing  
  _Fix:_ In the catch block, before `currentGame = null;`, run `try { if (typeof currentGame.unmount === "function") currentGame.unmount(); } catch (_) {}` so partially-acquired timers/listeners are released.
- **`js/registry.js:30`** — One missing/broken game module kills the entire registry and blanks the shelf  
  _Fix:_ Build defensively: `var GAMES = [Games.bomb, Games.mostlikely, ...].filter(function (m) { return m && m.meta; }).map(entry);` so a single failed module drops one card instead of the whole shelf.
- **`js/store.js:68`** — rosterGet validates only Array.isArray, not element shape — a corrupt entry bricks the roster screen unrecoverably  
  _Fix:_ Sanitize elements in rosterGet: keep only entries that are objects with a string name, and coerce bare strings, e.g. `return players.filter(function(p){return p && (typeof p === "string" || typeof p.name === "string");}).map(function(p){return typeof p === "string" ? { id: ..., name: p } : p;});`
- **`js/games/activity.js:451`** — Activity picks words with replacement — repeated words within one game are more likely than not, unlike every other content game's no-repeat queue  
  _Fix:_ Keep one shuffled queue per tier (built with Spielecke.shuffle in startGame, rebuilt when empty), and pop from it in pickWord — the same pattern used by nhie/mostlikely/whoami/truth.
- **`js/content/hochadel.js:247`** — "Der Dieb" trump lets a player steal a freshly drawn card, but the engine cannot transfer ownership — a stolen Gold card stays labeled with the victim's name forever  
  _Fix:_ Either add a holder-reassignment affordance for face-up trumps (small "pass to…" action per active card), or reword/restrict the Dieb card so it only applies to sofort effects the table can execute verbally.
- **`js/games/doodle.js:270`** — Doodle strokes drift away from the finger if the canvas is resized mid-draw (rotation, viewport/keyboard resize) — bitmap and pointer math are frozen at setup size  
  _Fix:_ Rescale pointer coords to the bitmap in pos(): x = (e.clientX - r.left - canvas.clientLeft) * (canvas.width / dpr) / canvas.clientWidth (same for y). Optionally also listen for window resize during the draw step and re-create the bitmap (preserving content via drawImage from a snapshot).
- **`js/games/doodle.js:272`** — Doodle drawing state is single-pointer: a second simultaneous touch corrupts the stroke with lines jumping between fingers  
  _Fix:_ Track the active pointer: on pointerdown, if an activePointerId is already set, return; else activePointerId = e.pointerId. In onMove/onUp, ignore events where e.pointerId !== activePointerId; clear activePointerId in onUp and teardownCanvas.
- **`js/games/imposter.js:296`** — Imposter can repeat the same secret word across rounds in one session (no used-word tracking)  
  _Fix:_ Maintain a per-mount used-word set or a shuffled queue keyed by the selected pools (mirroring whoami.js buildQueue/queuePool), rebuild it when exhausted or when the pool selection changes, and pop words from it in pickWord().
- **`js/games/maxchen.js:141`** — Mäxchen: lifting the hat while the dice are shaking orphans the roll timer, which later stomps the reveal screen  
  _Fix:_ In doReveal(), bail out or cancel the pending roll: `if (timer !== null) return;` (or `clearTimer();` first), and symmetrically ignore hat taps once mx-hat--lift is set (extend the guard at line 116 to also check 'mx-hat--lift').
- **`js/games/quiz.js:264`** — Quiz: level-clamped questions get a fresh used-map at every higher level, so hard questions repeat across rounds  
  _Fix:_ Key the used-set by question identity instead of (level, pool index), e.g. a single `used[q.q] = true` map checked in pickQuestion, or track used per category + that category's own clamped level index.
- **`js/games/liars.js:164`** — Liar's Numbers repeats questions within a session — no used-question tracking  
  _Fix:_ Keep a session-level set of used question texts (reset it only when every question in the resolved pools has been used, mirroring quiz.js pickQuestion), and clear it when the pool selection changes via the Pools().bind onChange callback.
- **`js/games/quiz.js:266`** — Quiz: pool-exhaustion reset allows the just-asked question to repeat immediately within the same round  
  _Fix:_ On reset, at minimum exclude the most recently asked index (and ideally all indices asked during the current roundQueue cycle) from the refilled `avail` when pool.length > 1.
- **`js/ui.js:30`** — Space key double-advances tappable prompt cards, silently skipping a prompt  
  _Fix:_ In tappable's keydown handler call e.stopPropagation() after preventDefault() so the shell's global Space shortcut does not fire a second, independent activation on the same key press (alternatively, have shell.onGlobalKeydown ignore the event when e.target itself is/contains a [data-primary] element).
- **`js/games/activity.js:169`** — Activity: stale persisted teamAssign after the roster shrank leaves one team empty, yet the game starts and gives the empty team turns  
  _Fix:_ In ensureAssignment, after dropping stale players, rebalance when one team is empty (or when |counts[0]-counts[1]| > 1) by moving the last-assigned players over; alternatively disable #act-start with a hint while either membersA or membersB is empty in renderSetup.
- **`js/games/bomb.js:271`** — No AudioContext.resume() anywhere — after any iOS interruption mid-round the ticks and the explosion are permanently silent for that round

Apply the same guard in whoami.js beep().
- **`js/games/whoami.js:202`** — Who Am I? countdown setInterval freezes when the screen auto-locks — the forehead pose means no touches reset iOS auto-lock for up to 90s  
  _Fix:_ Compute a wall-clock deadline (`var endAt = Date.now() + settings.roundSeconds * 1000`) and derive `remaining = Math.ceil((endAt - Date.now())/1000)` each tick so a lock/unlock jumps the clock correctly; call finishTurn() immediately if the deadline passed while hidden (visibilitychange). Request `navigator.wakeLock.request('screen')` at startTurn (try/catch, re-acquire on visibilitychange, release in stopCountdown/unmount) so the screen stays awake during the forehead pose.
- **`js/games/imposter.js:41`** — Imposter meta claims supportsDrinking but the game has no drinking mode at all  
  _Fix:_ Change imposter.js:41 to supportsDrinking: false (or, if a drinking mode is actually wanted, add the 🍻 toggle + drink resolution and update spec decision #2 to 15 games).

## Architecture & optimization plan

The 20 game modules were built independently over time and it shows: the same handful of UI patterns are hand-rolled in 5–12 places each, occasionally drifting into a bug when one copy gets fixed and its siblings don't (the id-vs-name keying bug fixed in Horse Race — commit `a44d157` — is still present in Truth or Drink). Ranked by lines saved × how many future games inherit it for free.

### 1. A shared "deck game" factory
_4 games duplicate this — nhie · mostlikely · truth · princess_

Never Have I Ever, Most Likely To, Truth or Drink, and Princess Treatment are the same game — setup screen, no-repeat draw queue, tap-card loop, drinking-mode toggle — with different content and one line of resolution text. A factory turns each into a ~15-line config object.

```js
Spielecke.deckGame({ id, name, icon, tagline, minPlayers, supportsDrinking,
  content: fn -> pools, field: "prompts",
  kicker: fn(t), rule: fn(t, settings) }) -> module
```

### 2. One no-repeat draw queue, used everywhere
_5 copies + missing in 7 games — fixes real repeat bugs_

"Draw without repeats until the pool is exhausted, then reshuffle" is reimplemented five times with subtle differences — and Imposter, Liar's Numbers, Rank It, Activity, and Quiz Out's level-clamped mode don't have it at all, so content visibly repeats within one sitting. A single bag primitive fixes six bugs at once.

```js
Spielecke.drawBag(buildFn) -> { next(fallback), reset() }
```

### 3. Setup-screen scaffolding
_~12 games rebuild this by hand_

Every game's setup screen assembles the same four pieces (roster note, pool chips, drinking toggle, start button) from scratch. A small helper module cuts each game's setup render to a handful of lines.

```js
Setup.rosterNote(roster, min, t)
Setup.drinkToggle(container, id, store, settings)
Setup.playAgainOrSetup(ctx, min, start, setup)
```

### 4. A chip-toggle binder
_7 games + ~10 hand-wired click loops_

The single-select "highlight active chip" pattern already has a well-built multi-select cousin in pools.js (Pools.bind). Generalizing it removes seven private copies of the same function.

```js
Spielecke.bindChoice(container, attrName, get, set)
```

### 5. "Pass the phone to {name}" screen
_5 games — Imposter · Wavelength · Liar's Numbers · Rank It · Doodle_

The privacy hand-off screen is identical in structure across five games with only copy differences.

```js
Spielecke.passScreen(container, { i, n, name, note, cta, onGo })
```

### 6. A shared sketchpad
_Doodle + Activity today — free for any future drawing game_

The pointer-events canvas is duplicated wholesale between Doodle Drama and Activity's draw round. It's also where two confirmed Doodle bugs live (canvas-resize drift, single-pointer corruption on a second touch) — fixing it once fixes both call sites.

```js
Spielecke.sketchpad(canvasEl, opts) -> { clear(), freeze(), toDataURL(), destroy() }
```

### 7. Drag-to-reorder list
_Rank It + Hochadel_

Vertical drag-reorder is hand-built twice with the same touch-drag math. Extracting it also opens the door to fixing the confirmed a11y gap (pointer-only, no keyboard path) in one place.

```js
Spielecke.dragList(listEl, { itemSelector, indexOf(el), move(from, to), onChange })
```

### 8. Consolidate on the one shuffle
_7 private Fisher-Yates copies despite a documented Spielecke.shuffle_

The shared, spec-mandated shuffle already exists in ui.js. Seven games quietly reimplement it instead — pure housekeeping.

### 9. A small audio kit
_Bomb + Who Am I?_

WebAudio setup/teardown/beep/vibrate is duplicated between the two games that use sound. Centralizing it is also the natural place to add the iOS AudioContext.resume() fix, so both games get it at once.

```js
Spielecke.audioKit() -> { beep(freq, dur, gain, type), close() }
Spielecke.buzz(pattern)
```

### 10. A dead-code sweep
_~25 dead i18n keys · 3 orphaned CSS blocks · unused 138 KB logo file_

Small on its own, but nine separate low-severity findings that all resolve the same way: delete. Worth one PR — i18n.js's dead German entries, main.css's orphaned Doodle chain-reveal block, cards.js's unused exports, the never-referenced spielecke_logo.jpg, Fuck the Dealer's dead timer.

## Suggested roadmap

Sequenced so each phase is shippable on its own and later phases build on earlier ones.

**Phase 1 — Fairness bugs (one evening).** Rank It's false "perfect match," Liar's Numbers' and Busfahrt's tie handling, Truth or Drink's infinite-loop freeze on duplicate names — small, isolated fixes. Horse Race's stale-bet replay follows the same fix shape as the Hochadel patch already shipped.

**Phase 2 — Reliability, "don't lose the round."** `history.pushState` + `popstate` in the router, a confirmation before header taps tear down a mounted game, a `beforeunload` guard. iOS: `AudioContext.resume()` on interruption, wall-clock deadline reconciliation for Bomb's fuse and Who Am I?'s countdown, `touch-action: manipulation` app-wide via `.tappable`.

**Phase 3 — Architecture consolidation.** Items 1–4 above (deck-game factory, draw-bag, setup scaffolding, chip binder) first — highest leverage, and each closes several medium-severity content-repeat bugs as a side effect. Then items 5–9 as time allows.

**Phase 4 — i18n, content, and spec cleanup.** Sweep hardcoded-English strings into i18n.js; fix `lang="en"` to track the language toggle. Batch the dead-code removal (item 10); reconcile PARTY-APP-SPEC.md against the code while this list is fresh.
